# 双仓库协作模式与接口变更规则

## 1. 目的

本说明用于固定 `NodeConsoleApp2` 与 `NodeConsoleApp2-SpineAssets` 的协作模式，使两个仓库可以并行推进，而不依赖同一个 CLI 长期手工协调。

## 2. 角色分工

### 2.1 主工程 `NodeConsoleApp2`

负责：

1. 战斗规则
2. 运行时事件语义
3. `presentationId` 的业务归属
4. 运行时展示适配层
5. 素材缺失时的降级策略

不负责：

1. `Spine Editor` 工程文件
2. Rig 制作
3. 动画导出

### 2.2 素材工程 `NodeConsoleApp2-SpineAssets`

负责：

1. 读取主工程角色目录
2. 管理分件图、Rig、动画和导出
3. 输出 `bundle_manifest / character_manifest / skeleton / atlas / textures`
4. 维护导出产物校验规则

不负责：

1. 写回主工程玩法数据
2. 直接修改主工程战斗规则

## 3. 镜像文档规则

以下主题必须在两个仓库各自有一份镜像文档：

1. 协作模式
2. 输入输出契约
3. 版本变更规则
4. 人工验收与自测入口

允许视角不同，但事实必须一致。

## 4. 共享对象与所有权

### 4.1 主工程拥有

1. `player / enemy / level` 玩法配置
2. `presentationId` 的业务来源
3. 战斗展示事件名和触发时机
4. 展示失败时的降级路径

### 4.2 素材工程拥有

1. `bundle_manifest` 目录结构
2. `character_manifest` 内容完整性
3. `Spine` 导出产物命名
4. 槽位命名与动画资源清单

### 4.3 共享拥有

1. `presentation_catalog` 字段语义
2. `schemaVersion`
3. 动画名与插槽名的运行时可见集合

## 5. 变更分级

### 5.1 非 breaking change

以下变更可单仓库先行，但必须在镜像文档中登记：

1. 新增可选字段
2. 新增可选动画
3. 新增不会影响旧消费者的注释字段

### 5.2 breaking change

以下变更视为 breaking change：

1. 删除字段
2. 修改字段语义
3. 修改目录结构
4. 调整必填项
5. 修改动画名、槽位名或资源寻址规则

breaking change 必须满足：

1. 在两个仓库分别建立 issue
2. 同轮更新镜像文档
3. 升 `schemaVersion` 主版本
4. 在主工程明确降级策略

## 6. 并行开发边界

在 `A2` 完成后，两个仓库可以并行，但前提是只围绕已冻结接口推进：

1. 主工程可以开发导入适配层、降级逻辑、展示驱动
2. 素材工程可以开发样本角色导入、Rig、导出与预览
3. 任一方不得私自扩写共享字段并假定另一方自动兼容

## 7. 验证要求

### 7.1 主工程侧

必须证明：

1. 即使没有任何 `Spine` 素材，主流程仍可运行
2. 导入失败时可回退到现有静态展示层

### 7.2 素材工程侧

必须证明：

1. 能从主工程同步 `presentation_catalog`
2. 导出产物能通过 manifest 校验
