# WBS L0：NodeConsoleApp2-SpineAssets 研发总纲

## 1. 总目标

建立独立的 `Spine` 素材制作工程，使其可以向 `NodeConsoleApp2` 稳定输出战斗展示素材产物。

## 2. WBS 主树

1. `1 工程底座与输入输出约定`
   - `1.1 项目底座建立`
   - `1.2 双仓库协作与交付接口`
   - `1.2.1 双仓库协作模式与变更流程`
   - `1.2.2 目录清单与交付清单版本化契约`
2. `2 样本回归验证`
   - `2.1 官方样本导入、Rig 与导出验证`
3. `3 正式生产主线`
   - `3.1 输入接收`
   - `3.1.1 request 输入契约与请求目录规则`
   - `3.2 任务建档`
   - `3.3 拆件加工`
   - `3.3.1 原图拆件与过程留痕`
   - `3.4 角色包收口`
   - `3.5 交付包导出`
   - `3.6 运行验收`
4. `4 主工程接入`
   - `4.1 主工程导入与交付规范`

## 3. 节点依赖

1. `1 -> 2 -> 3 -> 4`
2. `1` 负责建立独立仓库、输入输出约定和双仓库协作规则
3. `2` 负责证明样本 bundle 契约和导出验证链可持续回归
4. `3` 负责把正式生产主线固定为六个线性步骤：
   - `3.1 输入接收`
   - `3.2 任务建档`
   - `3.3 拆件加工`
   - `3.4 角色包收口`
   - `3.5 交付包导出`
   - `3.6 运行验收`
5. `4` 负责把正式导出物纳入主工程消费链

## 4. 当前活跃节点

1. `3 / 3.2`

## 5. 阶段关闭标准

1. `1`
   - 独立仓库可运行
   - 输入输出契约成文
   - 同步脚手架能读取主工程真实目录
2. `2`
   - 至少一个玩家样本和一个敌人样本完成导入、Rig、导出、预览验证
3. `3`
   - 六个主流程阶段都能被清楚地区分输入、处理和输出
   - `request / package / export` 三层契约稳定
   - 同一 `presentationId` 下多 `variant` 的交付语义明确
   - request 输入能在 package 内留下组件加工过程证据
   - variant 对组件的依赖可被机器判定为完整或不完整
   - 组件不完整时 package/export 会稳定拒绝导出
4. `4`
   - 主工程能够按约定消费 bundle，且失败路径可诊断
