## 1. 本清单是验证什么的

本清单验证的是“主工程与素材工程之间的交付文件契约是否已经冻结”。

它不验证：

1. 某个角色是否已经完成 Rig
2. 某个 bundle 是否已经真实导出
3. 主工程是否已经把这些 bundle 接入运行时

它只验证：

1. 三类文件的字段语义是否写清
2. 样例文件之间是否能构成最小自洽链路
3. 目录与命名规则是否足以支撑后续独立开发

## 3. 进入前说明

为了避免误解，先明确两点：

1. `schemaVersion` 是文件结构版本，不等于素材包内容版本
2. `bundleVersion` 是 bundle 发布版本，不等于字段结构版本

如果这两个概念没有被明确拆开，后续并行开发时就容易把“内容更新”和“结构破坏性变更”混为一谈。

## 4. 人工验收步骤

### 4.1 presentation_catalog 是否固定了业务到表现的桥

操作：

1. 打开 `presentation_catalog.schema.json`
2. 打开 `presentation_catalog.example.json`

检查重点：

1. schema 中必须存在 `meta / players / enemies`
2. 实体必须包含 `id / name / presentationId / bodyParts / visual`
3. 样例中玩家与敌人都给出 `presentationId`
4. `visual.defaultSkin` 与 `visual.spineSkeletonId` 允许先为 `null`

为什么这能证明：

1. `presentation_catalog` 是从主工程玩法配置走向素材工程的桥，如果桥两端字段不清晰，后续导入和导出都无法稳定推进

### 4.2 bundle_manifest 是否固定了 bundle 级目录语义

操作：

1. 打开 `spine_bundle_manifest.schema.json`
2. 打开 `bundle_manifest.example.json`
3. 对照设计说明中的 `1. 交付目录` 与 `2. bundle_manifest 职责`

检查重点：

1. `bundle_manifest` 必须包含 `schemaVersion / bundleId / bundleVersion / generatedAt / sourceCatalog / characters`
2. `characters[].characterManifest` 必须指向 `characters/<presentationId>/character_manifest.json`
3. `sourceCatalog.schemaVersion` 必须能表达它来自哪个 `presentation_catalog` 结构版本

为什么这能证明：

1. `bundle_manifest` 是主工程定位单角色资源的入口文件，如果目录语义在这里没固定，后续导入层就会跟着漂移

### 4.3 character_manifest 是否固定了单角色运行时可见资源

操作：

1. 打开 `spine_character_manifest.schema.json`
2. 打开 `character_manifest.example.json`
3. 对照设计说明中的 `3. character_manifest 职责`

检查重点：

1. 必须包含 `presentationId / skeletonFile / atlasFile / texturePages / animations / slots`
2. `skeletonFile` 与 `atlasFile` 的命名应与 `presentationId` 一致
3. `animations` 与 `slots` 应体现主工程运行时真正能看到的集合
4. `anchorProfile / scaleProfile` 应明确存在

为什么这能证明：

1. 主工程运行时真正依赖的是这些文件名、动画名和槽位名，而不是素材制作过程的内部细节

【】

### 4.4 三类样例是否构成最小自洽链路

操作：

1. 对照阅读三个 example 文件
2. 重点检查：
   - `bundle_manifest.example.json`
   - `character_manifest.example.json`
   - `presentation_catalog.example.json`

检查重点：

1. `bundle_manifest.sourceCatalog.schemaVersion` 与 `presentation_catalog.meta.schemaVersion` 能对应上
2. `bundle_manifest.characters` 中至少有一项 `presentationId` 能对上 `character_manifest.presentationId`
3. 样例中的路径和命名不会互相冲突

为什么这能证明：

1. `A2.2` 验的不只是“文件都写了”，而是“这些文件组合起来时仍然说的是同一个交付协议”
