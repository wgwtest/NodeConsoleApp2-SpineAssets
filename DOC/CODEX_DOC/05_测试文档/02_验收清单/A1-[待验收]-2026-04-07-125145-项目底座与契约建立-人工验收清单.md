# A1 项目底座与契约建立 人工验收清单

## 1. 这份清单验证什么

这份清单用于确认“独立素材工程是否已经具备最小可启动能力”，不是验证最终动画效果。

它验证的是三个问题：

1. 这个仓库是否已经是独立工程、独立 git
2. 主工程与素材工程之间是否已经有可执行的输入契约
3. 这个契约是否已经能被真实主工程数据跑通


需要能访问主工程目录：

```text
/home/wgw/CodexProject/NodeConsoleApp2/NodeConsoleApp2
```

## 3. 验收步骤

### 3.1 确认独立工程与独立 git

执行：

```bash
git rev-parse --show-toplevel
```

通过标准：

1. 输出根目录应为 `NodeConsoleApp2-SpineAssets`
2. 不应落在主工程仓库内部

### 3.2 确认设计与计划入口齐全

阅读以下文档：

1. [工程总体分析](/home/wgw/CodexProject/NodeConsoleApp2-SpineAssets/DOC/CODEX_DOC/01_需求分析/00-工程总体分析.md)
2. [总体设计](/home/wgw/CodexProject/NodeConsoleApp2-SpineAssets/DOC/CODEX_DOC/02_设计说明/01-独立素材工程总体设计(spine_assets_architecture)-设计说明.md)
3. [输入输出契约](/home/wgw/CodexProject/NodeConsoleApp2-SpineAssets/DOC/CODEX_DOC/02_设计说明/02-主工程与素材工程输入输出契约(game_spine_contract)-设计说明.md)
4. [A1 研制计划](/home/wgw/CodexProject/NodeConsoleApp2-SpineAssets/DOC/CODEX_DOC/03_研制计划/02-WBS-A1-项目底座与契约建立-研制计划.md)

通过标准：

1. 能清楚看出本工程只负责素材工作流，不负责战斗规则
2. 能清楚看出与主工程的输入输出边界
3. 能清楚看出当前只完成到 `A1`

### 3.3 确认自动化测试可运行

执行：

```bash
npm test
```

通过标准：

1. 测试结果为通过
2. 用例中应包含 `--config` 相对路径解析行为验证

### 3.4 确认真实主工程同步可运行

执行：

```bash
cp config/source-project.example.json config/source-project.json
npm run sync:catalog
```

为什么这样验：

1. 这一步验证的不是“脚本能写文件”
2. 而是验证“独立素材工程能否按约定读取主工程真实数据，并生成自己的统一目录”

通过标准：

1. 控制台输出包含 `SYNC OK`
2. 生成文件：
   - `workspace/imports/game_catalog.json`
3. 生成结果中应能看到：
   - `players` 长度为 `1`
   - `enemies` 长度为 `14`

### 3.5 抽查输出文件语义

打开：

1. [workspace/imports/game_catalog.json](/home/wgw/CodexProject/NodeConsoleApp2-SpineAssets/workspace/imports/game_catalog.json)

通过标准：

1. `meta.source.gameRepoRoot` 指向主工程真实目录
2. `players[0].presentationId` 为 `char_player_001`
3. `enemies` 中能看到 `goblin_story_headhunter`
4. `visual.defaultSkin` 和 `visual.spineSkeletonId` 当前允许为 `null`
