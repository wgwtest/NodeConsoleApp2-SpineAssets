## 1. 本清单是验证什么的

本清单不是验证动画效果，也不是验证主工程已经接入 `Spine`。

它只验证三件事：

1. 只打开一个仓库时，是否也能看懂本仓库负责什么、不负责什么
2. 是否已经明确写清 breaking change 的双仓库联动流程
3. GitHub 协作面是否已经有镜像节点，便于两个 CLI 或多人并行推进

如果这项通过，说明后续两个仓库可以在不共享同一会话上下文的前提下继续协作。

## 4. 人工验收步骤

### 4.1 单看素材工程文档能否读懂职责分工

操作：

1. 打开素材工程协作模式设计说明
2. 阅读 `2. 角色分工`
3. 阅读 `4. 共享对象与所有权`

检查重点：

1. 能清楚读到主工程负责战斗规则、事件语义和降级策略
2. 能清楚读到素材工程负责 Rig、动画、导出和 manifest 完整性
3. 能清楚读到双方共享的只有 `presentation_catalog` 语义、`schemaVersion` 和运行时可见集合

为什么这能证明：

1. 如果职责边界在单仓库文档里仍然读不清，就无法支持独立 CLI 或多人并行推进

### 4.2 主工程镜像文档是否与素材工程一致

操作：

1. 打开主工程镜像协作文档
2. 对照阅读 `2. 主工程职责`、`3. 主工程消费什么`、`5. 降级规则`

检查重点：

1. 主工程只消费正式导出产物，不消费 `.spine` 源文件
2. 当 bundle 缺失、schema 不兼容、单角色 manifest 缺失时，都写清了回退规则
3. 这些事实与素材工程文档不冲突

为什么这能证明：

1. `A2.1` 不是只把一边写清，而是要求两边对同一事实保持镜像一致

### 4.3 breaking change 联动流程是否足够具体

操作：

1. 回到素材工程协作模式设计说明
2. 阅读 `5. 变更分级`
3. 对照主工程文档中的 `6. 双仓库变更规则`

检查重点：

1. 是否明确哪些改动属于 breaking change
2. 是否明确要求两个仓库分别建 issue
3. 是否明确要求同轮更新镜像文档
4. 是否明确要求升 `schemaVersion` 主版本
5. 是否明确由主工程承担降级或迁移策略

为什么这能证明：

1. 只有把 breaking change 的动作链写成显式步骤，协作模式才不是口头约定

### 4.4 GitHub 协作面是否已有镜像节点

操作：

1. 打开素材工程 issue `#5`
2. 打开主工程 issue `#72`
3. 打开两个仓库的 Project

检查重点：

1. `#5` 与 `#72` 都存在
2. 素材工程 Project 中能看到 `#5`
3. 主工程 Project 中能看到 `#72`
4. 这两个节点都能被识别为当前双仓库协作基线的一部分

为什么这能证明：

1. `A2.1` 的目标是让后续工作不依赖聊天历史继续推进，所以必须在 GitHub 协作面留下稳定入口
