# 工程总体分析

## 1. 为什么单独建仓库

本工程独立出来的直接目的有两个：

1. 让 `Spine` 素材工作流不污染主游戏工程
2. 在输入输出契约稳定后，让主工程与素材工程能够并行推进

补充目标：

1. 把“角色表现资产生产”从“战斗规则实现”中剥离
2. 让素材试错、Rig 调整、导出校验不影响主工程主流程
3. 在后续建立远端仓库后，允许素材工程独立排期、独立提交、独立验收

## 2. 上游输入

当前默认读取主工程的以下目录数据：

1. `assets/data/config.json`
2. `assets/data/player.json`
3. `assets/data/enemies.json`

## 3. 下游输出

本工程未来默认输出：

1. `presentation_catalog`
2. `Spine bundle manifest`
3. `Spine skeleton / atlas / texture pages`

## 4. 非目标

当前仓库初始化阶段不做：

1. 完整 `Spine` 运行时集成
2. 正式美术生产
3. 大规模角色拆分
4. 主工程 battle render backend 改造

## 5. 当前阶段的最小交付

当前 `A1` 节点只交付最小可启动基线：

1. 独立 git 仓库和文档根
2. 主工程输入目录到素材工程中间目录的同步脚手架
3. `presentation_catalog` 与 `bundle_manifest` 的首版契约
4. 首轮自测、验收清单和交接记录

当前 `A2` 节点补充交付：

1. 双仓库协作模式规则
2. `presentation_catalog / bundle_manifest / character_manifest` 版本化接口定义
3. 主工程消费边界与素材工程交付边界的镜像文档

## 6. 后续阶段依赖关系

后续推进遵循“先契约、再样本、后接入”的顺序：

1. `A1`
   - 把输入输出边界和启动方式固定下来
2. `A2`
   - 把双仓库协作模式、接口归属、变更流程和版本规则固定下来
3. `B1`
   - 在固定边界内做样本角色的分件、Rig、导出验证
4. `C1`
   - 由主工程消费导出产物，并建立版本化导入规范

这意味着两个工程可以并行，但并不是无条件并行：

1. `A1` 完成前，只适合做低耦合准备
2. `A1` 完成后，可以开始补双仓库协作规则与版本契约
3. `A2` 完成后，主工程和素材工程可以围绕同一份契约并行推进
